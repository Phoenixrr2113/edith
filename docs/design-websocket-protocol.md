# WebSocket Protocol: Device–Cloud Communication

**Issue:** CLOUD-WS-047
**Status:** Design
**Depends on:** #48 (device auth — `lib/auth.ts`), #118 (state sync — `TAURI-SYNC-118`)

---

## Overview

The Tauri desktop companion (see `docs/desktop-companion.md`) communicates with the Edith cloud orchestrator via a persistent WebSocket connection. This document defines the full protocol: authentication handshake, message types, heartbeat, state sync, task dispatch, offline queuing, and reconnection.

The connection is **bidirectional**:
- **Cloud → Device**: agent state updates, messages to display, sync payloads, progress events
- **Device → Cloud**: user input, screen context summaries, sync requests, pings

---

## Architecture Fit

```
Tauri App (device)                    edith.ts (cloud / Fly.io)
─────────────────                     ──────────────────────────
CloudTransport (client)  ←── WS ───→  CloudTransport (server)
  └─ offline queue                      └─ injectMessage() / dispatchToClaude()
  └─ reconnect loop                     └─ lib/session.ts (active query)
  └─ state machine                      └─ lib/tick.ts (sync emitter)
```

The server side (`lib/cloud-transport.ts`) is a WebSocket server mounted inside `edith.ts`. It bridges the WebSocket protocol to the existing `injectMessage()` / `dispatchToClaude()` dispatch engine. All message routing on the server side uses the interfaces already established in `lib/session.ts` and `lib/dispatch.ts`.

## Connection Establishment

### Endpoint

```
wss://<fly-app>.fly.dev/ws
```

Query parameter: `?deviceId=<deviceId>` (used for per-device registry)

### Authentication (depends on #48)

On WebSocket upgrade, the client sends a JWT in the `Authorization` header:

```
Authorization: Bearer <jwt>
```

The JWT is generated by `lib/auth.ts#generateDeviceToken(deviceId, DEVICE_SECRET)`:
- Algorithm: HS256
- Claims: `{ sub: deviceId, iat, exp }` — 30-day lifetime
- Signed with `DEVICE_SECRET` (Fly.io secret + Tauri keychain)

Server rejects the upgrade with `HTTP 401` if the token is missing, invalid, or expired. The client should refresh tokens when expiry is within 7 days (on reconnect).

### Handshake Sequence

```
Client                              Server
  │── HTTP Upgrade (+ Auth header) ──→│
  │                                   │ verifyDeviceToken()
  │←── 101 Switching Protocols ───────│
  │                                   │
  │←── { type: 'connected', ... } ────│  server sends welcome
  │                                   │
  │←── { type: 'sync', ... } ─────────│  server sends initial state sync (#118)
  │                                   │
  │  (connection established)         │
```

## TypeScript Interface Definitions

```typescript
// Base envelope — all messages share this shape
interface WsMessage {
  type: WsMessageType;
  id?: string;       // optional correlation ID for request/response
  ts: number;        // unix timestamp ms (sender's clock)
}

// ── Message Type Union ──────────────────────────────────────────────────

type WsMessageType =
  // lifecycle
  | 'connected'
  | 'ping'
  | 'pong'
  // cloud → device
  | 'message'
  | 'state'
  | 'progress'
  | 'sync'
  | 'error'
  // device → cloud
  | 'input'
  | 'screen_context'
  | 'sync-request';

// ── Cloud → Device ──────────────────────────────────────────────────────

/** Server welcome on successful connection */
interface WsConnectedMessage extends WsMessage {
  type: 'connected';
  deviceId: string;
  serverVersion: string;
}

/** Edith message to display in speech bubble */
interface WsTextMessage extends WsMessage {
  type: 'message';
  text: string;
  from: 'edith';
  /** If true, message should trigger "talking" character animation */
  speak?: boolean;
}

/** Character/agent state update — drives Rive animation state machine */
interface WsStateMessage extends WsMessage {
  type: 'state';
  state: AgentState;
}

type AgentState =
  | 'idle'       // nothing running
  | 'thinking'   // dispatch started, no tools yet
  | 'working'    // tool calls in flight
  | 'talking'    // sending response to Randy
  | 'sleeping';  // night mode (22:00–08:00)

/** Background agent progress — drives "working" state details */
interface WsProgressMessage extends WsMessage {
  type: 'progress';
  taskId: string;
  description: string;
  /** 'started' | 'progress' | 'complete' | 'failed' */
  status: 'started' | 'progress' | 'complete' | 'failed';
  lastTool?: string;
  toolUses?: number;
  durationMs?: number;
  summary?: string;
}

/** Full state sync payload (see #118 — TAURI-SYNC-118) */
interface WsSyncMessage extends WsMessage {
  type: 'sync';
  payload: SyncPayload;
}

interface SyncPayload {
  schedule: ScheduleEntry[];
  taskboard: string;           // raw markdown from ~/.edith/taskboard.md
  contacts: ContactEntry[];    // first 50 by recency
  settings: Partial<AppSettings>;
  syncedAt: number;            // unix ms
}

interface ScheduleEntry {
  id: string;
  label: string;
  cronExpression: string;
  nextFireAt: number;          // unix ms
}

interface ContactEntry {
  name: string;
  identifier: string;
  lastSeenAt: number;
}

interface AppSettings {
  quietHoursStart: string;   // "22:00"
  quietHoursEnd: string;     // "08:00"
  characterVisible: boolean;
  speechBubblesEnabled: boolean;
}

/** Server-side error */
interface WsErrorMessage extends WsMessage {
  type: 'error';
  code: WsErrorCode;
  message: string;
  fatal?: boolean;  // if true, client should not reconnect immediately
}

type WsErrorCode =
  | 'AUTH_FAILED'
  | 'AUTH_EXPIRED'
  | 'RATE_LIMITED'
  | 'INTERNAL_ERROR'
  | 'BAD_MESSAGE';

// ── Device → Cloud ──────────────────────────────────────────────────────

/** User text input from keyboard or voice transcription */
interface WsInputMessage extends WsMessage {
  type: 'input';
  text: string;
  source: 'voice' | 'keyboard';
  /** Tauri deviceId — server validates against JWT sub */
  deviceId: string;
}

/** Gemini screen context summary — injected as ambient context */
interface WsScreenContextMessage extends WsMessage {
  type: 'screen_context';
  summary: string;
  apps: string[];        // e.g. ["VS Code", "Terminal"]
  confidence: number;    // 0–1 from Gemini
}

/** Device requests a full sync payload (e.g. after settings change or stale cache) */
interface WsSyncRequestMessage extends WsMessage {
  type: 'sync-request';
}

// ── Heartbeat ────────────────────────────────────────────────────────────

interface WsPingMessage extends WsMessage {
  type: 'ping';
}

interface WsPongMessage extends WsMessage {
  type: 'pong';
}

// ── Discriminated union helpers ──────────────────────────────────────────

type AnyWsMessage =
  | WsConnectedMessage
  | WsTextMessage
  | WsStateMessage
  | WsProgressMessage
  | WsSyncMessage
  | WsErrorMessage
  | WsInputMessage
  | WsScreenContextMessage
  | WsSyncRequestMessage
  | WsPingMessage
  | WsPongMessage;
```

## Heartbeat / Keepalive

- Client sends `{ type: 'ping', ts }` every **30 seconds**
- Server responds with `{ type: 'pong', ts }` (echoing the same `ts`)
- If the client receives no `pong` within **10 seconds** of a `ping`, it treats the connection as dead and begins reconnection
- Server closes connections that have been silent (no ping, no message) for **90 seconds**

```
Client                    Server
  │── ping (t=0) ────────→│
  │←── pong (t=0) ────────│   (within 10s)

  [30s gap]

  │── ping (t=30000) ─────→│
  │←── pong (t=30000) ─────│

  [client sends ping, no pong within 10s]
  │  reconnect()           │
```

## State Sync Messages (depends on #118)

State sync is **cloud → device, one-directional**. The device is a read-only cache.

**When sync occurs:**
1. On initial connection (server emits `sync` immediately after `connected`)
2. Every 15 minutes while connected (server-side timer per connected client)
3. On `sync-request` from device (e.g. after settings change, or stale cache on launch)
4. After any scheduled task completion that modifies taskboard

**Server-side emission point:** `lib/tick.ts` — on each scheduler tick, if a task completed, emit updated sync payload to all connected WebSocket clients.

**Stale check:** On Tauri app launch, if last sync timestamp is >30 minutes old, send `sync-request` before rendering UI.

## Task Dispatch Messages

When the device sends `{ type: 'input', text, source }`, the server routes it through the existing dispatch pipeline:

```
Device input message
  → server receives WsInputMessage
  → if session active: injectMessage(text) via lib/session.ts
  → if no active session: dispatchToClaude(text) via lib/dispatch.ts
  → dispatch emits task events (task_started, task_progress, task_notification)
  → server maps task events → WsProgressMessage → sends to device
  → dispatch emits agent state changes → WsStateMessage → sends to device
  → Edith sends response via send_message MCP tool (Telegram)
    OR server intercepts and also sends WsTextMessage to device
```

**State transitions emitted to device:**

| Dispatch event | WsStateMessage.state |
|---|---|
| dispatch_start | `thinking` |
| first tool_use detected | `working` |
| task_started (background agent) | `working` |
| send_message tool_use | `talking` |
| dispatch_end | `idle` |
| quiet hours active | `sleeping` |

## Error Handling

### Protocol Errors

- **AUTH_FAILED / AUTH_EXPIRED**: server sends `WsErrorMessage` with `fatal: true`, then closes with code `4001`. Client should refresh token and retry.
- **BAD_MESSAGE**: server sends `WsErrorMessage` with `fatal: false`. Connection stays open.
- **INTERNAL_ERROR**: server sends `WsErrorMessage`, connection may close. Client reconnects normally.
- **RATE_LIMITED**: server sends `WsErrorMessage` with `fatal: false`. Client backs off for the duration indicated in the message.

### Close Codes

| Code | Meaning | Client action |
|---|---|---|
| 1000 | Normal closure | Do not reconnect unless user action |
| 1008 | Policy violation (auth) | Refresh token, then reconnect |
| 4001 | Auth failed | Refresh token, then reconnect |
| 4002 | Token expired | Refresh token, reconnect immediately |
| 4003 | Rate limited | Wait 60s before reconnect |
| 1011 | Internal server error | Reconnect with backoff |

## Reconnection

The device is responsible for reconnection. The server is stateless per-connection.

**Algorithm: exponential backoff with jitter**

```
attempts:  1    2    3    4    5+
delay(s):  1    2    4    8    30  (cap at 30s)
jitter:    ±0.5s on each attempt
```

**Offline queue:** While disconnected, the device queues `WsInputMessage` and `WsScreenContextMessage` payloads locally (in-memory, max 50 entries, FIFO drop on overflow). On reconnect, the queue is flushed in order before any new messages.

**On reconnect:**
1. Re-authenticate (check token expiry, refresh if needed)
2. Send queued messages in order
3. Send `sync-request` if last sync is stale (>30 min)
4. Resume normal operation

```typescript
// Reconnection state machine
type ConnectionState =
  | 'disconnected'
  | 'connecting'
  | 'authenticating'
  | 'connected'
  | 'reconnecting';
```

## Server Implementation Notes (`lib/cloud-transport.ts`)

The server should:

1. Mount a WebSocket handler at `/ws` inside `edith.ts` (Bun's native `Bun.serve()` with `websocket` handler)
2. Maintain a `Map<string, ServerWebSocket>` registry keyed by `deviceId`
3. On upgrade: call `verifyDeviceToken()` from `lib/auth.ts` — reject with 401 if invalid
4. On `input` message: call `injectMessage()` if session active, otherwise `dispatchToClaude()`
5. Hook into `lib/dispatch.ts` event emissions to forward `task_started / task_progress / task_notification` → `WsProgressMessage` to all connected devices
6. Hook into `lib/tick.ts` to emit `WsSyncMessage` on connect and on task completion
7. Forward `WsScreenContextMessage` summaries to the active session as ambient context via `injectMessage()`
